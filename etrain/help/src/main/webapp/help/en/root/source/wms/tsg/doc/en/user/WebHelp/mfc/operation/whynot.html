Why not?

Purpose

This topic explains a general approach to find out why a transport unit does not move in the way it is supposed to. Along with the analysis of the issue the typical fault recovery actions are mentioned.

The techniques described here are not meant to be a world formula which can explain all problems conceivable in a warehouse. It is intended to be a starting point for a project specific trouble shooting guide which can address the special peculiarities of a certain installation.

As with source code debugging, there is no single way to analyze problems. Trouble shooting is always subject to personal preferences.

 CAUTION 

In any case, please make sure that you have a clear understanding of the cause of the problem prior to carrying out and fault recovery actions. Otherwise the situation might get even worse.

Not Moving

The trigger of this scenario is that there is somewhere a transport unit in the warehouse which is supposed to move but doesn't. The task is to find out why this is the case.

The starting point of the analysis is the MFC transport order dialog: identify the transport order of the suspicious transport unit via the barcode.

The further analysis strongly depends on the status of the transport order.

 

Status

 Trouble shooting steps

sent

This status means that the processing within the transport subsystem has successfully been completed and a corresponding mission should exist in the PLC.

  • Check the Message Trace whether a such a mission (a TUMI telegram) exists. If yes the problem is beyond the scope of the MFC. Please consult your contact person on the PLC side.
    If the PLC needs a telegram to be re-sent, this can be done by using the "back to partial source" action or by resending the telegram out of the Message Trace.
  • We have to find out why the mission has not been sent to PLC. The most likely reason is that we have no connection to the PLC. You can verify this via the PLC connection state dialog or by looking at the local visualization system (WinCC).
    Assuming a correct network parameterization in the MFC, there might be a connection problem via the network or due to a PLC being switched off.
  • If there is a valid connection the cause of the problem is a crash in the mapping within the message manager. However, after commissioning this is very unlikely.
    Check the syslog using the name of the corresponding message manager as the filter criterion.

rejected

This status means that MFC has no plan how to move the transport unit forward.

  • The most likely cause is that there is a problem in the routing configuration.
    A typical example would be, that a new transport unit type has been introduced in the system and that this new type is not yet configured for the relevant connections.
    The last entry in the TU-Log of the transport unit should point to the cause of the problem. The typical message is ".. no default hop found ... from .. to ...".
    After the configuration problem has been analyzed and resolved the transport order can be continued by using the "Resume transfer order" action.
  • Another cause might be a crash in the strategy of a transport subsystem. In order to avoid hick-up effects, transport orders causing a crash are also updated to status rejected.
    Consult the "Modify user" column of the transport order or the transport order archive to find out what subsystem caused the crash. In addition to this note the timestamp of the crash. Having this information in place, the syslog or the trace file can give more insight in the cause of the problem.
    In general, a software fix is required to recover from such a situation. After having received a patch the transport order can be continued by using the "Resume transfer order" action.

created

This status means that the transport order has not been completely processed by the global dispatcher. This situation rarely happens in a productive environment. Potential reasons are:

  • The global dispatcher is not up and running. One can verify this by using the system control dialog. This dialog can also be used to restart DbTM applications if necessary.
  • A crash within the global dispatcher. Consult the syslog or the trace file of the global dispatcher to find out details.

active

This status means that a transport subsystem has decided not to sent a mission to the PLC. In this case the reason why the transport unit does not move lies within MFC. The following activities can be done to find out why:

  • Verify whether there exists a concrete reason why the transport unit should not move, whether the decision of the MFC is correct or not. A typical example is that a transport does not start due to a capacity constraint on the global destination or due to sequencing.
    The last entry in the TU-Log of the transport unit should point to the cause of the problem.
  • Find out the transport subsystem the transport order is currently assigned to and consult the syslog or the trace file.

no to found

Since the MFC relies on transport orders, it is not surprising that a transport unit without transport order does not move.

Check following to find out why there is not transport order:

  • If the transport point the transport unit is currently located on is an automated I-point, the PLC might not have send a destination request. This can be verified by the Message Trace. If no destination request exists, please consult your contact person on the PLC side.
    If a destination request exists, either the superimposed host system had a problem to create a transport order or the MFC had a problem in handling the result. Consult the syslog or the trace file of the relevant transport subsystem to find out details.
  • Consult the transport order archive whether a transport order existed in the past for this transport unit. Find out whether the transport order has been manipulated previously by a fault recovery action (use 'FLTR*' as filter criteria for the field "Message Id" in the syslog dialog).
  • If you cannot find a hint to a transport order within the MFC please check the transport requests in LMS, check the corresponding ERP table if a third party host is involved or consult your contact person on the host side.

In any case an internal transport order can be used to move the transport unit out of the way.

 

 

      
Wrong Destination

There are a lot of regular reasons why a transport unit does not arrive at locations where it was supposed to go. Example: a transport order is redirected to the clearing station due to contour problems. In general, these kind of problems are easy to be understood for the user, especially as a reasonable error message  should show up when the transport unit arrives at the clearing station.

The following approaches can be taken to analyze non-regular situations.

  • Consult the  transport order archive to find out which transport subsystem changed the global destination of the transport order and when. Having this information you can use the syslog or the trace file of the relevant transport subsystem to find out details.
  • Consult the Message Trace to find out whether there are an irregularities originating from the PLC. Typical indicators are unexpected destination requests or TUEX telegrams.
  • Most likely, the TU-Log won't give valuable information in irregular situations, but can be consulted nevertheless.
