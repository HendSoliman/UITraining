Transports

Purpose

The screen Transports provides transport information of all levels with consideration to:

  • LMS (Logistics Manager) requests a transport execution at the underlying transport execution system (MFC) using Transport Requests.
  • MFC (Material Flow Controller) executes the transports represented by Transport Orders.

 INFO 

For troubleshooting operations in the global transport arena, see (for MFC related topics:) Material Flow Management Errors and (for LMS related topics:) Material Flow Management Errors in the Trouble Shooting Guide.

Special Buttons

Button

Description

Create an Operator Transport.
Required fields:

  • Type (Inbound, Outbound, Relocation, Clearing)
  • Load Unit (Load Unit moved by this Transport Request)
  • Initial Destination Location (typically the Virtual Location of a desired Area or Aisle )

If specialties are required, refer to fields described in Transport Detail.

Restart Transport

  • An existing (MFC) Transport Order is booked back to its initial source location.
  • Transport Request is reset, re-released and submitted to the underlying transport system (MFC) for execution.
  • Valid status of the Transport Request must be: "Error" (Default for manual clarifications).

Complete Transport

  • Transport Request is manually finished on current Destination Location.
  • Valid status of the Transport Request must be: "Error" (Default for manual clarifications) or "active".
  • To complete a Transport Request with status "Error", the Load Unit must reach the Default Destination. Otherwise, it is possible to use, "Restart Transport"  function.

Cancel Transport

  • The Transport Request is cancelled while submitting a Cancellation Request to the underlying transport system (MFC).
  • If the Transport Order of the MFC can cancelled, the Transport Request is subsequently cancelled.
  • Otherwise, the Transport Request stays in current status and the Transport Order of the MFC must be clarified within the MFC.
  • Valid status of the Transport Request must be: "Error" (Default for manual clarifications), "released" or "active".

 INFO 

 Ensure that the related Task of this Transport Request can be cancelled from a business case point of view. If the superior Task deals e.g. Order execution (e.g. marshalling of an Order or Tour) the Load Unit of this cancelled Transport Request is to be considered for further processing of this Order).

Release and Submit Transport Request

  • Transport Request is re-released and submitted to the underlying transport system (MFC) for execution.
  • With the release, a new transport destination is initially determined (see Putaway Strategy Administration)
  • If the Transport Order within MFC is not yet active (or not existent), the Transport Request is released for execution (status "released").
  • If the Transport Order within MFC is yet active, the Transport Request is set to "Error" status and the Transport Order of the MFC must be clarified within the MFC.
  • Valid status of the Transport Request must be: "released".

Menu for Fault Recovery of one selected transport:

 INFO 

These functions are very low level and provide only a limited validation. Often they cannot do any validation at all, since they are intended to be used to eliminate inconsistencies which might cause a validation to fail. Thus, make sure you know what you are doing when using this functionality. Improper use might cause severe damage to the integrity of the whole system. As rule of thumb, after the end of the fault recovery the PLC data and the MFC data must match again.

Example: When using the "Arrive at" function make sure by visual inspection that the transport unit is really physically located at the corresponding transport point on the conveyor system.
Failure to do this can lead to painful mismatch situations, especial for tracked conveyor systems.
  • Locking / unlocking of a transport order
    A locked transport order is not processed further by the MFC. Used to "park" faulted transport orders until final clarification.
  • Resume a transport order
    Recovers rejected transport order after the cause of the rejection (e.g. a missing connection) is fixed. Technically, functionality is identical to an "Arrive at" at the current location of the transport unit.
  • Back to partial source
    Books the transport unit back to the partial source to the connection it is currently traveling on. Consequently, the transport telegram for the actual connection repeats. The execution of this task is delegated to the strategy of the current transport subsystem.
  • Confirm current connection
    Finishes the current connection and places the transport unit on the destination of this connection. The transport order is continued from this transport point normally.
  • Arrive at Location ...
    User can specify the location the transport unit shall be placed on. The transport order continues from this new location. If the given location is equal to the global destination of the transport order, the user is asked if the transport shall be finished. If there are no possible connections between the global source of the transport order and the new given location, an error message displays.
  • Back to global source
    Transport is re-started from its global source.
  • Change global destination
    User can specify a new location to use as the global destination for the transport order. If there are no possible connections between the current location of the transport order and the new given destination, an error message displays. The transport order is not redirected on the fly, i.e. the global destination change takes effect only after the next routing, which is triggered when arriving at the next notification point.
