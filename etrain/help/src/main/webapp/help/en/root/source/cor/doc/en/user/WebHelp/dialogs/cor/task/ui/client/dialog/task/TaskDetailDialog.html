Detail Dialog Task

Purpose

This screen opens when viewing a Task. 

      
Screen

      
Fields

Field

Description

ID

Unique ID of the Task.

Task Type

Overall type of Task and the assigned required resources. For each booking or operational Task in the system, a special Task type is provided (e.g. Clearing, Operator Retrieval, Replenishment, Stocktaking, Supplier Advice, etc.)

Urgency

Basic sequence of automatic fulfillment tasks. Lower value indicates previous task execution. Urgency value is calculated by the system. Calculation factors are project specific and dedicated to the specific task type and can be changed using the appropriate policy.
Usually, priority and delivery time are considered for shipping orders. For all other order types the system considers priority and latest start time for calculation of urgency.

Release Mode

How a Task will be released:

  • manually (Released manually by screen)
  • automatically (Released directly)
  • scheduled (Released based on a workflow management schedule. Must be defined project specific)
  • triggered (Released based on specific events. Must be defined project specific)
  • dependency on start (Released on start of first dependent Task)
  • dependency on complete (Released on completion of last dependent Task).

Status

Status

Current execution state of the Task:

  • create (Task was created and not ready for pending (in standard only for Orders in "prepared" status for subsequent processing before it becomes "pending"))
  • pending (Task is awaiting release for execution)
  • release (Task is released for execution)
  • active (Task execution started)
  • complete (Task execution finished and the Task is fully completed)
  • cancelled (Task was cancelled without the defined execution)
  • error (Task was aborted due to any error situation).

Error Reason

Free text for detailing the reason for the status error only.

Cancellation Reason

Reason for cancellation of this Task. Only filled for cancelled Tasks:

  • by host (Cancelled automatically by the system)
  • by user (Manually cancelled by a user).

Times

Deadline

Timestamp when the Task must be completed.

Start Time

Timestamp at which Task execution began.

End Time

Timestamp at which Task execution completed.

Reference

Owner

Subsystem that created this Task:

  • Transport Management (Management of all Transport Requests to the underlying transport subsystems (MFC, FLS))
  • Order Management (Management of all Orders with all possible order types in the system)
  • Receiving Management (Management of all (advice based) Receiving processes in the system).

Reference

Used by other subsystems during Task creation. E.g. Order IDs for reference.

Parent

Superior Task of this Task. Defines the parent Task within the Task hierarchy chain.

Dependencies

If the Task is dependent on others. These Tasks must be completed before this Task can be executed.

Dependents

If the Task has dependent Tasks. These Tasks cannot start until this Task completed its execution.

Children

If the Task has children. Children are subordinate Tasks in the Task hierarchy chain.
