Description

Process: Order Cancellation & Abort

Purpose

This topic describes the mechanisms in the Logistics Manager that applies in the order cancellation process.

 INFO 

The abort process is different to the cancellation in terms of the already picked parts which will be shipped anyway.
The differences are noted separately below.

The following configuration topics are used in this process:

  • orp-order:orderClient.maxStatusForCancellation
  • orp-order:orderClient.cancellationOfHostOrdersAllowed
  • orp-order:orderCustomer.isCancellationAfterLoadingAllowed.
      
Description

Step

Description

1.

The system checks whether Order Cancellation/Abort is allowed by checking the Control Data parameters described above.

2.

If Sortation Orders have been delegated to the Sortation System a Cancellation Request is sent.

 INFO 

Since the relation between Order and Sortation Order is highly project specific it might be required to change the Service OrderProcessor.cancelOrder.

3.

If the Order has a Wave assigned the Wave is unassigned from the Order.

4.

If Route determination for the order has been done through an external Transport Management System a cancellation information is sent to the TMS depending on the appropriate configuration.

 INFO 

Project specific extension is necessary!

5.

The following process is executed for each Order Line separately. In principle the following different conditions of an Order Line are considered specifically:

  • 5.1: Load Unit (Pick Container or Full Load Unit Retrieval) of the Order Line is already loaded, shipped or confirmed (to the external Host)
  • 5.2: Load Unit of the Order Line is already (partially) picked (or retrieved) but not yet loaded, shipped or confirmed
  • 5.3: Order Line is not yet picked (or retrieved).

5.1

Load Unit of the Order Line is already loaded, shipped or confirmed:

An Order Line having Full Load Unit Retrievals or Pick Order Lines in status "complete" will be left undone if the reservation status of a related Load Unit is "Load Unit confirmed", "Load Unit loaded" or "Load Unit shipped".

5.2

Load Unit of the Order Line is already (partially) picked (or retrieved) but not yet loaded, shipped or confirmed.

 INFO 

In case of abort: The first steps regarding the rollback of the reservation will be skipped.

The Reservation Status of the considered Load Unit of the Order Line is "Load Unit picked":

  • The Inventory Units of the Pick Container (or Full Load Unit Retrieval) that are assigned to the current Order Line changes from "picked" to "free" inventory.
  • The "Dispo Quantity" and the "Current Quantity" of the Order Line are adjusted accordingly.
  • Associated Dispatch Load Units are deleted in order to prevent loading.
  • All existing transports stay unchanged.
  • The Order Line is set to "cancelled".

     INFO 

    In case of abort: The Order Line status remains unchanged.
  • Cancellation Reason is set to "Operator" in case of system initiated cancellation and set to "Host" in case of Host requested cancellation.

 INFO 

Putaway operations for cancelled Full Load Unit Retrievals and for Pick Container are not initiated automatically since this is highly project specific. Not all possible current Locations of Load Units may serve as Source Locations of transports. These Load Units should be stored manually at the appropriate Inbound stations of the warehouse.

5.3

Order Line is not yet picked (or retrieved):

  • For all Full Load Unit Retrievals (of the Order Line):
  • Inventory Unit Reservations of the Full Load Unit Retrievals are unreserved.
  • The status of Full Load Unit Retrievals are set to "cancelled".
  • Transport Requests are tried to be cancelled.
  • The "Dispo Quantity" and the "Current Quantity" of the Order Line is adjusted accordingly.
  • For all Pick Order Lines (of the Order Line):
  • Pick Order Lines in status "pending" are set to "cancelled".
  • For Pick Order Lines in status "released" or "active" a Cancellation Request is sent to the Pick Controller.
  • When the Pick Controller notifies about successful cancellation:
  1. Inventory Unit Reservations of the related Pick Order Lines are unreserved.
  2. Transport Requests are tried to be cancelled.
  3. If the Current Base Quantity of the Pick Order Line is greater than zero the status of the Pick Order Line is set to "complete" otherwise to "cancelled".
  4. The "Dispo Quantity" and the "Current Quantity" of the Order Line is adjusted accordingly.
  5. The status of the Pick Order Line is set to "cancelled".
  • The Order Line is set to "cancelled".
  • Cancellation Reason is set to "Operator" in case of LMS initiated cancellation and set to "Host" in case of Host requested cancellation.

6.

The destination of all Dispatch Load Units of this Order is deleted.

 INFO 

Skipped in case of abort.

7.

If the Order is assigned to a Load the Order is unassigned from the Load and all Dispatch Load Units that are assigned to cancelled Order Lines are removed from the Load.

 INFO 

Skipped in case of abort.

8.

The Order status is set to "complete" when all Order Lines are in status "complete" or "cancelled".

9.

After changing the Order to "complete":

  • If the owner of the Order is "Host" the status is set to "to confirm" and a confirmation is sent to the host.
    If the confirmation was successful the status is set to "finished".

  • Otherwise it is directly set to status "finished".

10.

When the Order status is set to "finished" the Inventory Management is notified about the finish of the Order.
(Shipped Inventory Units are deleted while cancelled parts are reset to "free" inventory.)

 INFO 

Attention: In case of only parts of Dispatch Load Units are cancelled, these parts need to be removed from the Load Unit organizationally.

 INFO 

Skipped in case of abort.
