Operation

Purpose

This topic explains the how the Dematic MFC is operated.

 

      
Overview

In a sunny day scenario the MFC does not require any user interaction in order to perform its everyday life tasks. It just receives transport orders from the superimposed system, carries them out and confirms them back to the host.

In case of faults the MFC provides dialogs to analyze problems on the transport level. Typical questions are: "Why is box xyz not moving?" or "Why has box xyz been redirected to destination abc?". Such error diagnostics is possible for life data (i.e. still open transport orders) as well as for historic data (i.e. transport orders which have already been completed, e.g. with a wrong destination).

There are also mechanisms to recover faulty transport orders.

 INFO 

Most of the techniques described in this context are most important during the commissioning phase when both MFC implementation and PLC implementation are not yet completely settled. Thus, this information is especially relevant for service personnel.

General Approach

Please keep the following principle in mind when recovering from fault situations:

  • Tray to get a thorough understanding of what went wrong (esp. when there are complaints like "nothing works anymore"). No actions should be taken until it is clear what happened. The tools provided therefore are described in the chapter Error Diagnostics.
  • Error recovery means to resynchronize the physics with the data. This resynchronization always has to happen bottom up: PLC --> MFC --> superimposed host system.
    In this chain the lower system report the result of their actions to the higher ones.
  • Be aware that the clarification might no be complete when only the MFC level is re-synchronized.
  • Although the MFC tries to be as fault tolerant as possible, the MFC never triggers critical recovery actions automatically. The final decision on error recovery is always upto the end user.

 

 DANGER 

The improper use of fault recovery actions can cause severe damage to the integrity of the whole system since these actions can only perform a limited plausiblization.
Thus, please make sure you know what you are doing when using this functionality!!!

 

      
Error Diagnostics

The MFC provides the following screens to display transport level information which can be used for analyzing problems.

 

  • Show currently active transport orders (and transport units)

These dialogs show the status of all transport orders currently being active within the MFC (i.e. which have not yet been confirmed to the superimposed host system).

 INFO 

The partial source of a transport order is always the transport point at which the PLC has most recently reported the transport unit. Please be aware, that this transport point may differ from the current location of the transport unit.

For each transport order the corresponding TU-Log is shown (see next paragraph for further information). The last log entry is also shown in the (upper) table of the transport order. This information is especially helpful to find out why a transport unit is not moving.

The "Transport Order" dialog is the starting point for the fault recovery (see below). Besides that, the user can only change the priority of a transport order. Click here to find more information on the "Transport Orders" dialog.

For the vehicle subsystem and the multishuttle subsystem there are special variants of the transport order dialog which show all transport orders currently being active in a certain aisle combined with the status information of that aisle. This gives the end user the possibility to see at one glance what is going on in an aisle. Please refer to the online help for further information (vehicle).

 

  • Transport unit log

The transport unit log reflects the lifecycle of a transport unit while being active within the MFC, i.e. from the creation of a transport order until it is eventually confirmed back to the superimposed host system. The transport unit log is kept after the transport order completes. Thus, it can be used to analyze historic movements.

Since a concrete transport unit can be move more than once within the warehouse an additional identifier TU-key is introduced. Using this key it is possible to distinguish between individual transports of the same transport unit.

Whilst being quite technical, the transport unit log is fully internationalized and thus should also be readable to the experienced end user.

 INFO 

To make this log really useful, it is important that also project specific extensions of the core business functionality reflect their decisions in the log. Click here to find out how to use the Logistics Log API to create an TU-Log entry.

The TU-Log can be accessed via the menu "MFC --> System --> Transportunit Log". For active transport orders the log is shown below the transport orders table.

 

  • Equipment Status and PLC connection state

This dialog shows - in individual tabs - the current status of the transport equipment as it has been reported by the PLC. In general, the granularity of this information less detailed as it is shown in a visualization system (e.g. WinCC). Click here for further information on this dialog.

 INFO 

The most detailed information about the equipment status can always be retrieved from the visualization system.

In addition to displaying of pure status information, this dialog also offers the possibility to lock and to unlock individual equipment elements, e.g. an aisle or a connection. See below for further details.

The "PLC connection state" dialog shows the whether the MFC has a network connection to the subordinate PLC. Click here for further information on this dialog.

 

  • Transport Archive

Each insert and update operation of the central entities transport orders and transport units is archived in the database. This information is accessible via the menu "MFC --> Archive". The archive data is kept after the transport order completes. Thus, it can be used to analyze historic movements.

The "Modify User" column denotes the DbTM name of the transport subsystem which triggered the change in the database. Together with the timestamp this information can be used to find the trace file which holds the related business transaction.

Since this information is very technical it is not intended for the end user in the first place, but more for service personnel.

 

  • Message Trace

All telegrams which have been sent to the PLC resp. which have been received from the PLC are logged in the message trace. The log always contains the original raw data which have been exchanged, i.e. no further processing has happened. A double click on trace line shows the individual fields of the telegram in a better readable format.

 INFO 

If there are typical message sequences which deviates significantly from the standard flow of control or if the standard telegram formats have been tailored, it is important to tailor the online documentation of this dialog as well. This makes life easier for the service personnel.

Click here for further information on this dialog.

When working with the simulation where one has a mirrored message manager, all telegrams appear twice since they are log by both message manager instances.

 

  • Syslog

The MFC writes syslog messages to document important decisions and error situations. These messages can are displayed in the "System Log" dialog. Click here for further information on this dialog. Typical examples for syslog messages are:

  • process start / end
  • connection established / lost to PLC
  • error messages from PLC
  • locking / releasing  an aisle, group, connection,
  • other administrative actions in the transport system, e.g. fault recovery
  • rollback of a business transaction

 

  • System Controller Dialog

As for all DbTM applications the background application of the MFC can be monitored by the "System Controller" dialog. Click here for further information on this dialog.

If a background application has crashed, it can be restarted via this dialog. The reason for the crash can be found in the trace file of the application.

The size of the input queues can be used to get a clue about the current workload of an application.

 

      
Fault Recovery

The fault recovery mechanisms can be used to correct error situation on the transportation level. They also can be used to eliminate irregularities between the MFC point of view and the PLC point of view.

 

 DANGER 

The techniques described in this chapter are very low level and thus can only provide a limited plausiblization. Often they cannot do any plausiblization at all, since they are intended to be used to eliminate inconsistencies which might cause a plausiblization to fail.

Example: When using the "Arriva at" function make sure by visual inspection that the transport unit is really physically located at the corresponding transport point on the conveyor system. Failure to do this can lead to painful mismatch situations, especial for tracked conveyor systems.

 

 INFO 

The techniques described in this chapter only correct error situation on the transportation level. To completely clarify an error situation, additional actions on level of the superimposed host system might be necessary. In addition to that, correction also might be necessary at the PLC level.

The order of fault recovery in general bottom up: PLC level --> MFC level --> WMS level

A typical scenario in this context is the clarification of bins.

 

For better tractability, fault recovery actions are logged in the system log.

 

 

  • Recovery of transport orders

Transport orders can be recovered by first selecting one or more transport orders in the "Transport Orders" dialog and then by opening the context menu.

 INFO 

The fault recovery functionality requires the special access right "Transport Orders/Fault Recovery".

The following functionality is offered:

  • Locking / unlocking of a transport order
    A locked transport order will not be processed any more by the MFC. This functionality can be used to "park" faulted transport orders until they can be finally clarified.
  • Resume a transport order
    This functional can be used to recover rejected transport order after the cause of the rejection (e.g. a missing connection) has been fixed. From a technical point of view the functionality is identical to an "Arrive at" at the current location of the transport unit.
  • Back to partial source
    This functionality books the transport unit back to the partial source to the connection it is currently traveling on. As a consequence, the transport telegram for the actual connection is repeated. The execution of this task is delegated to the strategy of the current transport subsystem.
  • Confirm current connection
    This functionality finishes the current connection and places the transport unit on the destination of this connection. The transport order is continued from this transport point normally.
  • Arrive at Location ...
    The user can specify the location the transport unit shall be placed on. The transport order is continued from this new location. If the given location is equal to the global destination of the transport order, the user is asked if the transport shall be finished. If there are no possible connections between the global source of the transport order and the new given location, an error message is displayed.
  • Back to global source
    The transport is re-started from its global source.
  • Change global destination
    The user can specify a new location which shall be used as global destination for the transport order. If there are no possible connections between the current location of the transport order and the new given destination, an error message is displayed. The transport order is not redirected on the fly, i.e. the change of the global destination takes only effect after the next routing which is triggered when arriving at the next notification point.
  • Delete transport order
    This functionality deletes the transport order at its current location. A host confirmation is sent with error code "ABORT_DELETED", if necessary.
  • Confirm transfer order
    The transport order is finished at its global destination. A positive host confirmation is sent, if necessary.
  • Confirm transfer order at a different location …
    The user can specify a location at which the transport order is finished. A positive host confirmation is sent, if necessary.
  • Cancel transfer order
    The transport unit is placed on the global source of the transport order and the transport order is deleted without further plausiblization. A positive cancellation message is sent to the host, if necessary.

 

 

  • Creating internal transport orders

Transport orders can also be created on MFC level to initiate transport without interaction of the superimposed host system. If required, such transport orders can also be confirmed to the host. Click here for more information creating internal transport orders.

A typical use case is also the generation of transport orders for testing during the commissioning phase.

 INFO 

Internal transport orders do not take care about bin occupation or bin booking (either the MFC is configured to do so). In general however, this is up to the host.

A special case is the creation of retrieval orders for bin clarification which is possible using the transport unit dialog. This functionality is only available when the MFC is confgired to perform bin booking.

 

 

  • Resending PLC telegrams

The message manager trace can be used to re-send PLC level telegrams in both directions. Click here for further information on this dialog.

This functionality is rarely required, since the resending of telegrams can also be trigged by the recovery actions of transport orders, e.g. a "back to partial source" triggers the re-sending of the corresponding transport telegram. The using the transport orders recovery actions is more convenient and secure and should always be preferred.

In the following scenarios the resending of PLC telegrams are useful:

  • The MFC logic handling a PLC telegram is flawed but no rollback happened so that the PLC telegram is lost.
  • Special non-standard telegrams are involved where the standard fault recovery functionality cannot be applied to.

 

 

  • Resending DbTM telegrams

The DbTM messages dialog can be used to resent internal DbTM telegrams which are in state "locked" because they caused a rollback. Click here for the online help of the dialog.

 

  • Sortlane Clearance

If a sortlane sequencing is jammed up, it can be cleared by using the ZUSO monitor.

 

      
Examples

The chapter gives some example which show how the fault recovery mechanisms described above can be used in typical error scenarios.

 

  • Booking of movements carried out at the sub-ordinate level.

Due to a fault at the PLC level a transport unit has been moved at the subordinate levenl to another location within the warehouse . No PLC telegram could be sent which might reflect this subordinate movement.

The "Arrive At Location..." functionality can be used to re-synchronize the MFC. The transport will be continued from this new location.

 

  • Transport unit has been removed form the PLC.

A transport unit has broken down on and cannot be moved any more. The PLC level mission is deleted at the operators panel (together with any potential tracking information). In general, this is signaled to the MFC by an event code 'TD'.

 INFO 

The standard MFC functionality does not trigger any action upon 'TD' but just reject the transport order. This gives the end user the flexibility to decide on a per case base how to recover from the situation (e.g. it is possible to reconstruct the transport unit and continue from an induct station).

 

In case the transport unit cannot be recovered, the user can select the transport order within the MFC GUI and has to delete it manually.

Special variants thereof are sending a confirmation which signals the hosts that the box has arrived at the destination or sending a cancellation which signals that the box is still at the global source.

 

  • Recovery after a flawed configuration within the MFC

The host sent a transport to the MFC, but due to a flawed configuration the MFC has set the status of the transport order to rejected.

After having corrected the problem, the transport order can be resumed by using the "Resume a transport order" functionality which resets the transport order into status active.

 

  • Recovery after an software update during commissioning

This scenario describes techniques to continue a transport order which has been interrupted due to a software flaw. No general strategy can be given, but the following rules of thumb apply:

If the arrival telegram could by handled correctly, but the subsequent transport mission could not be sent anymore, use the fault recovery action "back to partial source" to trigger a re-sent.

If the arrival could not be booked correctly, e.g. important project specific tailored functionality has not been processed correctly. This has to be rectified first (according to the principle "synchronize PLC and MFC"). One approach might be to resent the PLC telegram or to release the internal DbTM telegram (which is in status locked in case of a rollback).

 

  • Redirection to clearing station

Transport orders can be redirected to the clearing station by using the fault recovery action "Change global destination".

 

  • Prevent transport units from moving on

A transport unit can be stopped during its journey by locking the transport order. This might be especially useful when the transport order is still in the outbound queue of a crane subsystem, i.e. has not yet begun to move.

 

  • Clearance of bogus UFO transport orders

Due to irregular telegrams having been received from the PLC, the MFC might generate bogus UFO transport orders, i.e. UFO orders which do not correspond to a physical transport unit. The transport orders can be removed from the MFC database by the fault recovery action "Delete transport order".

No host confirmation will be sent, since UFO orders are MFC internal transport orders.

 

      
Standard Operations

Besides error recovery the MFC provides the following functionality to control center personnel.

 

  • locking of transport orders

The locking of transport orders is part of the fault recovery functionality (see above).

 INFO 

Be aware that locked transport unit of locked transport orders won't move on and thus might block the material flow in the warehouse.

 

  • locking of transport equipment

Locks can be set on the following devices and entities: means of transport, connections, transport points, aisles and multishuttle levels. Locks are set and released by using the "Status" dialog.

 INFO 

In this context, a lock always related to the transport level and not to the dispositive level. The means that locking an aisle does not mean that the host does no longer send transport unit to that aisle. It only means that for this aisle no transport movements are carried out.

 

  • creating positioning movements

Positioning movements can be used to move the LHD of stacker crane or a multishuttle to a certain position without moving the fork.

For stacker cranes movement orders orders are created using the "Stacker crane" dialog of the corresponding aisle. For a multishuttle subsystem the "Status" dialog is used.

 

      
Database Cleanup

The following database tables of the MFC contain historic data and thus are infinitely growing if not purged periodically.

  • Message Trace (table MM_TRACE)
    See here for further information.
  • KPI data (tables TAR0X_*)
    See here for further information.
  • Transport archive (tables TS_AR_TRAORDER and TS_AR_TRAUNIT)
    See here for further information.
  • Logistics Log (table LOGISTIC_LOG)
    See here for further information.

 

The demo system has already preconfigured database clean-up with reasonable default values for purging these database tables. Click here to find out how this configuration can be changed.

The cron job PurgeTraOrderJob allows the purging of stale transport orders. Examples might be: UFO transport orders or transport orders for transport units which didn't show up after a certain amount of time.
